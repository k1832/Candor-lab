## 2. P20 — measured numbers and proposed pre-registration targets

**Machine caveat.** Intel Core Ultra 7 265F, 20 cores, Linux; release build of the
Rust *reference* compiler; warm OS page cache; wall-clock, best-of-3. All measured
projects are **below** the p20.rs reference scale (M≈50k lines / N=200 modules): the
largest here is the assembled selfhost tree at **19,345 lines / 9 modules**. Sub-100ms
timings carry meaningful process-spawn/timer overhead. Numbers are order-of-magnitude
anchors, not the reference-scale numbers the ratified gate will use.

### Measured (median of 3), current toolchain

| Operation | corelib (~1.8kL, 13 mod) | selfhost tree (19.3kL, 9 mod) | wasm_interp (2.0kL, 1 file) |
|-----------|--------------------------|-------------------------------|-----------------------------|
| `check` (no cache) | 0.013 s | **0.104 s** | 0.043 s |
| `build` cold (cache cleared) | 0.015 s | **0.116 s** | — |
| `build` warm no-op (all reused) | 0.004 s | 0.004 s | — |
| `build` incremental, one leaf body edit | 0.009 s | 0.033 s (edit codegen.cnr, 3.6kL leaf) | — |
| `build` incremental, root-module comment edit | — | 0.007 s (lexer.cnr; signature unchanged → **zero downstream re-check**, the P20 two-hash dividend) | — |
| `compile` → native dev (Cranelift) | 0.082 s | — | — |
| `compile` → native release (LLVM -O2) | 0.238 s | — | — |

Analysis throughput ≈ **185k lines/s** through `check` (19.3kL / 0.104s).

**C-toolchain baseline** (for the clean-build-multiple limb): `cc -O2` on a
comparable ~2000-line / 400-function C program = **0.668 s** (`-O0` = 0.131 s).
Candor's release native compile of a similar-size corelib is **0.238 s ≈ 0.35× the
`cc -O2` baseline** — i.e. currently *faster* than the C toolchain; dev compile
(0.082 s) beats `cc -O0` (0.131 s).

### Proposed pre-registration targets (PROPOSALS — for the deciding authority to ratify)

Grounded in the measurements above with headroom for reaching reference scale
(50k lines) and real-toolchain overheads. Expressed against the p20.rs reference
project (N=200 modules, M≈50k lines):

1. **T1 — incremental rebuild (warm cache, one module-body edit) ≤ 1.0 s.** Measured
   proxy: a 3.6kL module edit in a 19.3kL tree = 0.033 s; even the largest single
   module in a 50kL project stays far under 1 s. Headroom ≈ 30×. (P20's own text says
   "single-digit seconds"; the data supports the tighter ≤ 1 s — the authority may
   prefer ≤ 3 s for margin.)
2. **T4 — clean `check` of the reference project ≤ 3.0 s.** Measured throughput 185k
   L/s → 50kL ≈ 0.27 s; ≤ 3 s gives ~10× headroom for per-module and real-toolchain
   costs. (Withdraws gracefully to a "single-digit seconds" clean check.)
3. **Clean native `--release` build ≤ 2× the `cc -O2` baseline** for an equivalent-
   size program. Measured 0.35× today — large headroom under a 2× ceiling.
4. **T2 — downstream re-analysis on a body-only edit = 0** (already asserted green in
   `stage_c.rs` and re-measured by p20.rs). Ratify as a hard release criterion: this
   is the mechanism P20 rests on, and it is the cheapest to regress silently.

### What "in CI" requires

**There is no CI.** No `.github/`, no CI config of any kind exists in the lab. Both
gate items 1 and 2 are blocked on standing up CI first. Minimum: a workflow that (a)
runs the Rust suite, (b) runs `p20.rs` at reference scale and fails on target
regression, (c) runs the 0018 dispatch campaign, (d) runs the NN#16 reproducibility
test, and (e) once editions exist, the edition-migration gate. The distribution repo
(publication step 2) is the natural home; the lab may host the gate workflow.

### The genuinely 1.0-blocking open obligations

- **OBL-MODULES-ARTIFACT** (P20, ch11 §§7–10) — implemented (measured working) but the
  incrementality machinery is unspecified normatively; couple its discharge to gate #2.

### Already-done-but-unmarked

- **OBL-ITER-BORROW** — the §1 summary still tiers it as a feature gate, but line 617
  records it DISCHARGED (region-free `for read x in read coll`, 2026-07-10, extended
  2026-07-14). *Ledger hygiene: mark discharged in the summary.*
- **NN#16 reproducibility** items — done (candor.lock relative sources + two-roots test).

### Deferrable-past-1.0 (one-line justification each)

- **OBL-CONTRACT** — gates `audit`/`assumed-proven`; `enforced` (the shipping level) is
  fully specified. Additive later via edition.
- **OBL-FFI** — gates FFI *content* (design 0011); the boundary *marker* has a SKELETON
  home (ch11 §10.1). Needed for a rich stdlib/self-hosting but not for language safety.
- **OBL-TEXT** / **OBL-TEXT-RESULT** — text-type budget beyond `[u8]`; additive, gated
  on P3 design. ROADMAP names it as gating self-hosting, not the stability promise.
- **OBL-GENERICS-CLOSURE, -STRATEGY, -TURBOFISH, -EFFECT** — expressiveness/trigger
  gates; each fires only on a *measured* basket regression. Shipped surface is complete
  without them.
- **OBL-ITER-PIN** — the `for` library-shape resolution mechanism + `Opt`'s normative
  shape; a spec-completeness gap, small, resolvable within-edition but not safety-
  blocking. *Recommend closing before 1.0 for normative completeness (S).*
- **OBL-FMT-REBORROW** — the shipped formatter (`compiler/src/real/fmt.rs`) does a
  *conservative token-level* reborrow collapse but is explicitly **not type-aware**
  (its doc comment says it "cannot tell a read borrow from a write borrow"); the
  obligation's type-aware `f(read b.*) → f(b)` rewrite is open. Deferrable (canonical-
  form nicety), small.
- **OBL-MINMAX-INTRINSICS** — re-scoped to reserved-but-unimplemented; trivial.

### Three judgment calls the deciding authority must settle (the crux of item 3)

The sources contain a real tension: VERSIONING/ch00 §3.4 name OBL-WINDOW, OBL-ALIAS,
OBL-CONSIST as hard "no-1.0-before-discharge" gates, **yet** P10 says the first stable
edition ships **no concurrency**, and the fault-window draft can only discharge the
*single-threaded* core because the concurrency half is vacuous without atomics. This
cannot be read strictly (that would require shipping atomics before 1.0, contradicting
P10). The coherent reading — which the fault-window draft already argues (§1.1–1.2) —
is that each gate is discharged **to the extent the shipping edition's semantics
require**, with the concurrency parts deferred, as a *bundle*, to the edition that
ships atomics. Ratifying that reading is a judgment call, not engineering:

- **J1 (OBL-WINDOW):** Accept the rigorous-informal single-threaded core as
  discharging NN#20 *for the concurrency-free edition*, defer O1–O5 with atomics, and
  rule on whether mechanization is required now or is post-1.0 preferred-not-required.
- **J2 (OBL-ALIAS):** This one is **not** vacuous — the shipping backends optimize
  unsafe code. Rule whether 1.0 requires the aliasing/optimizer-assumption spec written
  (rigorous-informal minimum) now, or whether the current backends' assumptions are
  narrow enough to defer. Likely genuinely 1.0-blocking; treat as real spec work (L).
- **J3 (OBL-CONSIST):** Rule that the degenerate no-concurrency composition (already
  discharged by the fault-window draft) satisfies the gate for this edition, deferring
  the full C/C++20 transcription with atomics — OR require the transcription now.

## 5. Honest distance-to-1.0 estimate

**≈ 5–8 work-slices of this session's size**, several of them authority-gated rather
than engineering-bound. The language *core* is in strong shape: the safety semantics
are specified and NORMATIVE-DRAFT, the coherence/dispatch campaign is green, the P20
mechanism demonstrably works (measured), reproducibility is done, and Bet 5 is
confirmed. The distance is concentrated in **process and infrastructure**, not core
design:

- **CI from zero** (~1 slice) — the single biggest blocker, gating two of four clauses.
- **Edition machinery actually exercised** (~1–2 slices) — today it is a validated
  `edition` field with only one legal value; the migrator that exists is the syntax
  migrator, not an edition migrator.
- **OBL-ALIAS spec** (~1 slice, if ruled 1.0-blocking) — the one non-deferrable spec debt.
- **Fault-window promotion + P20-machinery spec + P20 ratification + Bet-5 re-measure**
  (~1–2 slices combined).
- **Authority rulings** (J1/J2/J3, verdict recording, target ratification) — low effort,
  but they are on the critical path and cannot be self-served by engineering.

The strict-vs-scoped reading of the pre-stability tier (J1/J2/J3) is the pivotal
decision: under the scoped reading the ledger is within ~2–3 slices of clear; under a
strict reading OBL-CONSIST/-WINDOW-concurrency would pull atomics forward and push 1.0
out by a full concurrency edition. The sources favor the scoped reading; it needs
ratifying, not discovering.
